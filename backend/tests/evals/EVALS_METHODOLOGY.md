# Evals Methodology

This document describes the agentic eval framework for the autoqa-prompt-optimizer system.

---

## What Are These Evals?

The evals are an LLM-as-judge test suite that verifies the **quality** of each agent node's output, not just its correctness. Traditional unit tests assert that code runs without error; these evals assert that the _content_ of LLM-generated outputs meets defined quality standards.

Each scenario exercises one node in isolation with a fixed input state and judges the output across multiple weighted dimensions.

---

## Why Node-Level Evals?

The optimization pipeline is an agentic loop. End-to-end accuracy testing is expensive, slow, and difficult to debug when a node produces a subtly wrong output that compounds through later iterations. Node-level evals:

- Catch regressions in one node without running the whole pipeline
- Make failure modes explicit and reproducible
- Enable prompt changes to be validated cheaply before touching live data
- Provide a signal for refactoring: if a node's output quality is stable, its internals can change freely

## Scenario Structure

Each scenario is a YAML file with the following top-level fields:

```yaml
id: <unique-id>
description: "Human-readable summary of what this scenario tests"
node: <node_name>
failure_mode: <optional — too_broad | ambiguous | stagnant | already_good | ...>
expect_interrupt: <optional bool — for clarification scenarios>

rule:            # The evaluation rule passed to the node
  rule_id: ...
  rule_type: answer | trigger
  speaker: Agent | Customer
  evaluation_type: entire | first | last
  n_messages: -1 | <int>
  description: |
    <structured rule description>

rca_findings: |  # Pre-computed RCA, used by optimizer and clarification nodes
  ...

iteration_history:   # Accuracy history — controls stagnation/high-accuracy guards
  - {iteration: 1, description: "...", accuracy: 0.65, precision: ..., recall: ..., f1: ...}

conversations:   # Conversations the node receives as input
  - id: c1
    transcript:
      - {speaker: agent, msg: "..."}
      - {speaker: customer, msg: "..."}
    ground_truth: "Yes" | "No"
    prediction: "Yes" | "No"

judge:           # Evaluation criteria for the node's output
  dimensions:
    - id: <dimension_id>
      weight: <float — must sum to 1.0>
      prompt: |
        <Judge prompt. Use {output} as placeholder for node output.>
  pass_threshold: 0.70
```

### Key conventions

- **`{output}`** is substituted with the node's actual output before the judge prompt is sent.
- **`iteration_history`** length controls whether stagnation guards trigger in `prompt_optimizer` and `mid_loop_clarification`. Three identical entries → stagnant.
- **`conversations`** with `prediction` set drives the ground-truth map and confusion matrix used by nodes that need prior evaluation results.
- Clarification scenarios with `expect_interrupt: true` must seed `session_store` before running — see `test_clarification.py`.

## The Judge Framework

Each dimension is scored by a separate LLM call using `gpt-5.2` (or the `EVAL_MODEL` env var). Scores are floats 0.0–1.0:

| Score | Meaning |
|---|---|
| 1.0 | Fully meets criterion |
| 0.7 | Mostly meets it |
| 0.4 | Partially meets it |
| 0.0 | Does not meet it |

The weighted sum across all dimensions is computed; a scenario passes if `weighted_sum >= pass_threshold` (default 0.70).

### Retry logic

The judge runner retries each dimension up to 2 times with exponential backoff on transient `APIConnectionError` or JSON parse failures. This eliminates false negatives from network flakiness under load.

## Writing Good Judge Rubrics

These patterns reduce judge hallucination and variance:

### Label the output explicitly

The judge LLM sometimes hallucinates "no output provided" when the placeholder text doesn't clearly indicate what to evaluate. Use a clear delimiter:

```yaml
prompt: |
  The text below is the clarification question generated by the system.
  Generated question: {output}
  Does the question...
```

Alternatively surround with `---`:

```yaml
prompt: |
  The text below is the full audit output:
  ---
  {output}
  ---
  Does the audit above...
```

### Ask structural questions, not simulation questions

**Avoid:** "Would an evaluator using this description classify c1 as No?"
The judge cannot reliably simulate evaluation without running the description.

**Prefer:** "Does the description explicitly require two or more pieces of identifying information?"
This is answerable from the description text alone.

### Provide graded scoring for borderline outputs

Binary 1.0/0.0 rubrics make the result brittle to LLM phrasing variation. Use graduated scoring:

```yaml
prompt: |
  Score 1.0 if the audit explicitly flags that criteria evaluate customer behaviour rather than agent actions.
  Score 0.7 if it identifies customer-scope issues (even framed as "implicit vs explicit") without naming the speaker mismatch.
  Score 0 if it only discusses accuracy without any mention of customer/agent scope.
```

### Include context the judge needs inline

Don't expect the judge to infer what the "existing description" says. Quote it:

```yaml
prompt: |
  The existing description already states: "Agent uses empathy phrases such as 'I understand your frustration'".
  Generated question: {output}
  Does the generated question probe something beyond those criteria?
```

### Accept equivalent framings in strategy rubrics

LLM outputs vary in vocabulary. A rubric that requires "agent actions" will score 0 when the output says "agent behaviour" or "what the agent does." Accept synonyms:

```yaml
prompt: |
  Does the audit recommend changing criteria to measure what the AGENT does rather than what the CUSTOMER says or does?
  Score 1.0 for explicit agent-action criteria. Score 0.7 for any shift from customer to agent scope.
```

## Dimension Design Patterns

### For optimizer scenarios

| Dimension | Weight | Tests |
|---|---|---|
| `improvement_direction` | 0.25 | Is the rewrite structurally different from the original? |
| `functional_correctness` | 0.35 | Does the new description fix the identified failure case? |
| `generalisation` | 0.25 | Does the description work beyond the test conversations? |
| `format_compliance` | 0.15 | Are all required sections present (METRIC_NAME, PASS_CRITERIA, EXAMPLES)? |

### For RCA scenarios

| Dimension | Weight | Tests |
|---|---|---|
| `root_cause_accuracy` | 0.40 | Does the RCA identify the right root cause? |
| `actionability` | 0.30 | Does the RCA recommend something the optimizer can act on? |
| `non_hallucination` | 0.30 | Does the RCA only reference patterns visible in the provided conversations? |

### For alignment audit scenarios

| Dimension | Weight | Tests |
|---|---|---|
| `gap_identification` | 0.40 | Does the audit find the actual gap? |
| `strategy_clarity` | 0.35 | Is the recommended strategy concrete and correct? |
| `non_hallucination` | 0.25 | Does the audit avoid inventing gaps not evidenced by the data? |

### For clarification scenarios

| Dimension | Weight | Tests |
|---|---|---|
| `question_relevance` | 0.40 | Does the question target the actual ambiguity? |
| `non_redundancy` | 0.35 | Does it ask something not already answerable from the description? |
| `plain_language` | 0.25 | Can a non-technical QA manager understand it? |

## Test Infrastructure Notes

### Stagnation guard in clarification tests

`mid_loop_clarification` calls `interrupt()` (from `langgraph.types`) when it generates questions. In test context, `interrupt()` raises `RuntimeError("Called get_config outside of a runnable context")` instead of a proper `GraphInterrupt`.

**Workaround:** The node writes questions to `session_store` _before_ calling `interrupt()`. Tests seed the session with `session_store.add(session_id, {...})` then read questions from the store after the exception.

```python
session_store.add(session_id, {"current_phase": "evaluating"})
try:
    await mid_loop_clarification(state)
except Exception:
    session_data = session_store.get(session_id) or {}
    stored_questions = session_data.get("clarifying_questions", [])
```

### `session_store.update()` requires prior `add()`

`session_store.update()` is a no-op for uninitialised sessions. Always call `session_store.add(session_id, {...})` before running a node that writes to the store.

### TypedDict instances are dicts

`ClarifyingQuestion` is a `TypedDict` — `isinstance(q, dict)` returns `True` and `q.get("question_text")` works. No special handling needed.

## Running the Suite

```bash
cd backend
source venv/bin/activate

# Full suite
pytest tests/evals/ -v

# Single scenario
pytest "tests/evals/test_optimizer.py::test_optimizer_quality[opt-stagnant-001]" -v

# All alignment tests
pytest tests/evals/test_alignment.py -v
```

Set `EVAL_MODEL` in `.env` to override the judge model (default: inherits `OPENAI_MODEL`).

## Adding a New Scenario

1. Create `scenarios/<node>/<id>.yaml` following the schema above.
2. Choose dimensions that test the failure mode — one structural check, one correctness check, and one non-hallucination check is a good baseline.
3. Keep `pass_threshold: 0.70` unless the scenario is intentionally strict.
4. Run the scenario alone to calibrate: if it passes on the first run, the rubric may be too lenient; if it never passes, the rubric may demand too much.
5. Commit the YAML. The test runner discovers scenarios automatically from the directory.
